# 森林培育学智能问答教学系统 — 系统介绍文档

## 一、项目背景与定位

森林培育学（Silviculture）是林学专业的核心课程，涵盖造林、营林、森林抚育、林分经营等大量专业知识。传统教学中，学生在课后复习时面临以下痛点：

1. **教材内容庞杂**：涉及造林密度、立地评价、混交林营造、抚育间伐等数十个知识模块，查找困难
2. **答疑资源有限**：教师无法随时在线答疑，学生课后学习效率低
3. **知识碎片化**：同一知识点可能分散在多本教材的不同章节

本系统旨在构建一个**基于教材知识库的 AI 智能问答平台**，让学生随时提问、即时获得基于教材原文的专业回答，同时为教师提供教学数据分析和教材管理工具。

### 核心理念

- **严格基于教材**：所有回答必须来源于已上传的教材内容，不编造、不臆测
- **引用可追溯**：每条回答附带教材来源（书名、章节、页码），便于学生回查原文
- **领域专业化**：内置 500+ 林业术语同义词库，精准理解专业问题

## 三、功能模块详解

### 3.1 学生端 — 智能问答

#### 核心功能

学生打开系统首页即可看到简洁的搜索界面，输入问题后系统实时返回基于教材的专业回答。

**交互流程**：
1. 学生在搜索框输入问题（如"造林密度的确定原则是什么？"）
2. 系统通过 SSE（Server-Sent Events）返回问答事件流（`meta` / `token` / `done`）
3. 当前默认采用“质量优先”策略：完成检索、生成与质量校验后，再输出最终答案正文
4. 每条回答附带引用来源列表，标注教材名称、章节和页码
5. 学生可对答案进行"有用/无用"评价反馈

**技术亮点**：
- **质量优先输出**：答案在输出前经过本地完整性检查、接地性检查和必要重写，减少“先给错后改稿”
- **SSE 事件式交互**：前端通过 `meta/token/done` 事件接收结果，便于显示来源、耗时和最终答案
- **引用标注**：答案中以 `[1]` `[2]` 等脚注标记引用，对应具体教材段落
- **答案缓存**：相同问题 30 分钟内直接返回缓存，节省 API 调用

#### 问答流程技术细节

```
用户提问
  │
  ▼
[语言检测] ── 中文 / 英文自动识别
  │
  ▼
[意图分析] ── 7种问题类型识别
  │           ├─ 定义题："什么是X？"
  │           ├─ 分类题："X有哪些类型？"
  │           ├─ 方法题："如何做X？"
  │           ├─ 比较题："X和Y的区别？"
  │           ├─ 条件题："X的条件是什么？"
  │           ├─ 优缺点题："X的优缺点？"
  │           └─ 一般问答
  │
  ▼
[混合检索] ── 从教材知识库中检索相关内容
  │           （详见第四章检索策略）
  │
  ▼
[Prompt构建] ── 将教材片段 + 问题组装为提示词
  │             严格约束 LLM 基于教材回答
  │
  ▼
[LLM推理] ── 生成候选答案（教材约束）
  │
  ▼
[质量门控] ── 本地完整性检查 + 接地性检查
  │           必要时触发一次低温重写
  │
  ▼
[结果返回] ── SSE事件返回（meta + final answer）+ 来源标注 + 记录入库
```

### 3.2 教师端 — 管理后台

教师端需要管理员权限登录，提供四大功能模块：

#### 3.2.1 数据仪表板

- **核心指标卡片**：累计查询数、今日查询数、已上传教材数、覆盖城市数
- **趋势图表**：近 14 天查询量和访客趋势折线图
- **地区分布**：Top-10 城市访客分布柱状图
- **待改进问题**：差评率超过 50% 的问题列表（帮助教师发现知识盲区）
- **存储统计**：数据库大小、各表行数
- **数据清理**：一键清理历史数据、清空向量缓存

#### 3.2.2 教材管理

- **大文件上传**：支持 PDF、DOCX、DOC 格式，最大支持 100MB+ 文件
- **分块上传**：大文件自动分块传输，显示上传进度条
- **教材元数据**：可编辑标题、作者、出版社、年份、版本、语言
- **状态管理**：实时显示教材处理状态（上传中 → 处理中 → 已发布 / 错误）
- **重新处理**：支持单本或批量重新处理教材（更新分块和向量）
- **删除管理**：删除教材及其所有关联数据

#### 3.2.3 查询记录

- **历史记录**：分页查看所有学生提问，按时间倒序
- **详细信息**：每条记录展示问题、答案摘要、使用的模型、响应时间
- **热门问题**：按频率排列的 Top-10 高频问题
- **教材使用率**：各教材被引用次数排行

#### 3.2.4 模型配置

- **多模型支持**：可配置 OpenAI、DeepSeek、通义千问、Ollama、自定义 API
- **参数调节**：temperature（创造性）、maxTokens（最大输出长度）
- **RAG 开关**：启用/禁用向量检索模式
- **Embedding 配置**：独立配置 embedding 模型和 API Key
- **连接测试**：一键验证 API Key 和网络连通性
- **一键切换**：多套配置自由切换激活

### 3.3 认证系统

系统支持两种登录方式：

| 方式 | 适用角色 | 说明 |
|------|---------|------|
| OAuth 登录 | 学生/教师 | 通过第三方 OAuth 认证 |
| 本地登录 | 管理员 | 用户名 + 密码，由环境变量配置 |

## 四、核心技术：混合检索策略

系统的核心竞争力在于针对林业教材优化的**混合检索引擎**，结合关键词检索和向量语义检索两种模式。

### 4.1 两种检索模式

| 模式 | 配置 | 优势 | 适用场景 |
|------|------|------|---------|
| 关键词检索 | useRAG = false | 速度快、零额外 API 成本 | 中等规模教材库 |
| 混合检索（RAG） | useRAG = true | 语义理解更精准 | 大规模教材、复杂问题 |

### 4.2 关键词三路召回策略

系统采用三路并行召回，确保既精准又全面：

```
问题关键词提取
  │
  ├─▶ 精确召回（Strict）
  │    权重: 5.8 | 限制: 100条
  │    核心关键词的精确匹配
  │
  ├─▶ 短语召回（Phrase）
  │    权重: 3.4 | 限制: 160条
  │    短语和意图关键词的匹配
  │
  └─▶ 宽松召回（Broad）
       权重: 1.5 | 限制: 260条
       扩展关键词（含同义词）的宽松匹配
```

### 4.3 林业领域同义词库

系统内置 500+ 组林业术语同义词对，大幅提升专业问题的召回率：

```
造林密度 ↔ 林分密度、初植密度、种植密度、密度控制
立地质量 ↔ 立地条件、立地指数、地位指数、立地评价
抚育间伐 ↔ 间伐、疏伐、抚育采伐、透光伐
混交林   ↔ 混交造林、针阔混交、乔灌混交、混交方式
人工更新 ↔ 人工造林、植苗造林、直播造林
天然更新 ↔ 自然更新、飞播造林、萌芽更新
... （共计500+组）
```

### 4.4 BM25 评分算法

采用经典 BM25 算法进行关键词相关性评分：

- **k1 = 1.2**：词频饱和度参数
- **b = 0.75**：文档长度归一化参数
- 综合考虑词频（TF）、逆文档频率（IDF）、文档长度

### 4.5 向量检索（RAG 模式）

当启用 RAG 时，系统额外执行向量语义检索：

1. 将用户问题通过 Embedding API 转为向量
2. 与教材分块的向量计算余弦相似度
3. 取 Top-30 语义相关候选
4. 与关键词检索结果融合排序

### 4.6 多维重排序与去重

所有候选结果经过多维度重排序：

| 维度 | 说明 |
|------|------|
| BM25 评分 | 关键词频率和文档相关性 |
| 召回多样性 | 是否来自多个召回路径（加分） |
| 邻接块扩展 | 自动补充相邻文本块（权重 0.82） |
| 向量相似度 | 语义层面的相关性（RAG 模式） |
| Jaccard 去重 | 相似度 > 0.55 的段落判定为重复 |
| 来源多样性 | 同一教材最多取 2 条、同一章节最多取 2 条 |

### 4.7 检索兜底机制

为避免“召回链路全空”导致直接失败，系统在以下场景启用应急兜底检索：

- 关键词三路召回 + 向量融合后候选集为空
- 重排序与多样性筛选后 Top-K 结果为空

兜底实现为 `emergencyLikeFallbackSearch`：基于问题核心词做 SQL `LIKE` 粗召回并打分排序，返回可用片段，保证问答链路稳定性。

## 五、文档处理管道

### 5.1 处理流程

```
上传教材文件（PDF / DOCX / DOC）
  │
  ▼
[文本提取]
  ├─ PDF → pdf-parse 库提取文字
  ├─ DOCX → mammoth 库提取文字
  └─ DOC → word-extractor 库提取文字
  │
  ▼
[语言检测] → 自动识别中文/英文
  │
  ▼
[智能分块]
  ├─ 按章节标题切分（识别"第X章"、"X.X"等结构）
  ├─ 在段落、句子边界处自然切割
  ├─ 中英文采用不同的分块参数
  └─ 相邻块之间保留重叠文本（保证上下文连贯）
  │
  ▼
[向量化]（RAG 模式下）
  └─ 为每个文本块生成 Embedding 向量
  │
  ▼
[存储入库] → material_chunks 表
```

### 5.2 分块参数（按语言自适应）

| 参数 | 中文 | 英文 | 说明 |
|------|------|------|------|
| 目标块大小 | 860 字符 | 1500 字符 | 理想的分块长度 |
| 最小块大小 | 420 字符 | 800 字符 | 低于此值会与前块合并 |
| 最大块大小 | 1080 字符 | 1850 字符 | 超过此值强制切割 |
| 重叠大小 | 150 字符 | 220 字符 | 相邻块之间共享的文本 |
| 段落合并阈值 | 500 字符 | 900 字符 | 小段落合并为一块 |

### 5.3 Token 估算

- 中文：约 1.5 字符 / token
- 英文：约 4 字符 / token

## 八、LLM 多模型支持

### 8.1 已支持的模型厂商

| 厂商 | 典型模型 | API 地址 | 特点 |
|------|---------|---------|------|
| OpenAI | GPT-4, GPT-4-Turbo, GPT-3.5-Turbo | api.openai.com | 能力最强，成本较高 |
| DeepSeek | deepseek-chat, deepseek-coder | api.deepseek.com | 国内直连，性价比高 |
| 通义千问 | qwen-turbo, qwen-plus | dashscope.aliyuncs.com | 阿里云，国内稳定 |
| Ollama | 各种开源模型 | localhost:11434 | 本地部署，零成本 |
| 自定义 | 任意 OpenAI 兼容 API | 自定义 | 灵活扩展 |

### 8.2 推理方式

- 所有模型统一使用 **OpenAI 兼容的 Chat Completion API**
- 驱动层保留流式接口能力（`invokeLLMStreamWithConfig`）
- 问答主链路当前采用“先完整生成 + 质量校验，再通过 SSE 输出最终答案”的质量优先策略
- Embedding 接口独立配置，可使用不同模型

| 缓存层 | 容量 | TTL | 说明 |

| RAM 向量缓存 | 全量 | 10 分钟 | 教材向量内存缓存 |
